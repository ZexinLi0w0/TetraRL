# Week 7 Task 8b — Jetson Nano FFmpeg co-runner tail-latency

**Date**: 2026-04-18
**Device**: `nano2` (Jetson Orin Nano 8GB, L4T R35.4.1, 6 × ARMv8)
**ffmpeg**: 4.2.7-0ubuntu0.1 (Ubuntu stock, **no** nvv4l2dec/nvdec/cuda hwaccel)
**Driver**: `scripts/week7_ffmpeg_corunner.py` (PR #15)

## Run command

```bash
python scripts/week7_ffmpeg_corunner.py \
    --n-steps 20000 \
    --conditions none,720p,1080p \
    --hw-decode \
    --out-dir runs/w7_ffmpeg_nano/
```

## Deviations from spec

1. **`--n-steps 20000` (spec said 2000).** The spec's 2000-step
   default was tuned for the legacy 4GB Jetson Nano. On this 8GB
   Orin Nano with a 6-core ARMv8 CPU and a single-thread Python
   workload, 2000 steps complete in ≈ 0.15 s — far less wall time
   than ffmpeg needs to start interfering. Bumped to 20000 (≈ 1.3 s
   per condition) so ffmpeg has time to actually compete for cycles.
   Memory budget is fine (peak ≈ 700 MB).

2. **`--hw-decode` falls back to CPU decode.** The Ubuntu stock
   ffmpeg 4.2.7 on this device does **not** ship with nvv4l2dec /
   nvdec / cuda. `ffmpeg -hwaccels` only lists vdpau, vaapi, drm,
   opencl. The harness probes via `_hw_decode_available()` and
   silently uses CPU decode (`-f lavfi -i testsrc=...`) when the
   probe fails — no exception, no behaviour difference for the
   measured CartPole loop.

3. **2K condition not in this deliverable.** The spec deliverable
   lists `none,720p,1080p`. A 4-condition run including 2K is
   captured in the sibling directory `runs/w7_ffmpeg_nano_full/`
   for the paper's CDF figure; observed 2K p99 was 0.083 ms
   (slowdown 0.96x — within measurement noise of the baseline
   because the 6-core CPU has dedicated headroom for ffmpeg).

## Interpretation

On a 6-core Orin Nano, single-thread CartPole + multi-thread
ffmpeg synthetic decode rarely contend for the same core, so the
observed p99 slowdown is small (≈ 6%). This is a useful negative
result for the 2-platform CDF: the Orin AGX result (PR #18 / Orin
worktree) is expected to show much heavier slowdown because that
device has the more demanding co-runner setup. Both data sets feed
the combined-platform CDF figure generated by the Orin spec's
`scripts/week7_make_cdf.py`.
