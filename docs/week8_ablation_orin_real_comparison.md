to fall back to the Mac substitute path because Orin SSH was unreachable.

matrix on real Orin AGX (`platform=orin_agx`, hostname=`ubuntu`, port 8002).

**Methodology reference**: see `docs/week8_ablation_design.md` for the

## 1. What's identical and what's not

The W8 progress.md (PR #24) called this out explicitly: with the eval
runner's `_make_telemetry()` falling back to the Mac stub for any
`platform="orin_*"` config, only the wall-clock-derived metrics
(`wall_time_s`, `tail_p99_ms`) move when we change the host machine.
Everything else is bit-identical because the action sequence is
deterministic given the seed and the synthetic telemetry is
host-independent.

This is what we observed on the re-run too. The W9 PR #25 fix
(commit `c87257f`) added a `RuntimeWarning` so the runner now declares
loudly that it is using the Mac stub; that warning fired **once** at
the start of our re-run, exactly as the W9 fix intended. We document
this as expected behaviour, not a bug — see Section 4.

## 4. Significance shifts: what changed when timing got noisier

The qualitative pattern from PR #24 is preserved: `resource_manager`
remains `ns` on every metric (`p (tail p99) = 0.735`, vs 0.610 on Mac),
which confirms the W8 design doc Section 5.3 prediction that with
`dvfs_controller=None` (still true at runner.py:416) the
`_NullResourceManager` substitute is mechanically equivalent to the
real `ResourceManager` because no DVFS index actually goes anywhere.

What **did** shift, and is interesting to note for the paper writeup:

| arm | Mac p (tail p99) | Orin p (tail p99) | direction |
|---|---|---|---|
| preference_plane | 0.685 (ns) | 0.041 (sig) | Orin reveals an effect that Mac couldn't resolve |
| rl_arbiter | 0.151 (ns) | 0.003 (sig) | strongly significant on Orin |
| override_layer | 0.326 (ns) | 0.050 (borderline) | borderline significant on Orin |
| resource_manager | 0.610 (ns) | 0.735 (ns) | unchanged — confirms design prediction |

In other words, the Mac substitute was too noisy to surface tail-p99
differences between arms; on Orin, three of the four non-baseline arms
move tail-p99 significantly relative to the `none` baseline. This
directly validates one of the W8 design doc's open questions:
"on Orin we expect to see tail_p99_ms separations that the Mac jitter
masked".

## 5. Confirmation that `resource_manager` is still `ns` on real Orin

The W8 design doc (Section 5.3) predicted that the `resource_manager`
ablation arm would remain `ns` on real Orin **for as long as the eval
runner constructs `TetraRLFramework(dvfs_controller=None)`**. We
verified by re-reading `tetrarl/eval/runner.py:416`:

    return TetraRLFramework(
        ...
        dvfs_controller=None,
    )

This is unchanged from PR #24. The `_NullResourceManager` substitute
returns the same DVFS index choice as the real `ResourceManager` would,
but neither index is ever applied to hardware (no `DVFSController` is
wired in), so no measurable difference can arise. The `ns` result on
real Orin (`p (tail p99) = 0.735`) is the **expected and correct**
outcome.

A future `feat(week10-orin-dvfs-runner): wire DVFSController into eval
runner` PR would be required to change this prediction.

## 6. W9 #25 RuntimeWarning behaviour

The W9 PR #25 fix (`c87257f`) added a `RuntimeWarning` in
`tetrarl/eval/runner.py:336–342` that fires when `_make_telemetry()` is
called with `platform.startswith("orin_")` because the real
`TegrastatsDaemon` is wired up only by the platform-specific scripts
under `scripts/`, not by the runner. Our re-run captured exactly one
emission of this warning at the start of the sweep (Python deduplicates
identical warning messages by default):

    runner.py:406: RuntimeWarning: _make_telemetry() called with
    platform='orin_agx' but the real tegrastats daemon is not wired up
    here; falling back to the Mac stub. Real-Orin runs should use the
    platform-specific scripts that build a TegrastatsDaemon directly.

This is the W9 fix doing its job: declaring the telemetry-source
fallback loudly so a downstream reader is not misled into thinking
`mean_energy_j` and `mean_memory_util` are real Orin readings. They are
not — they are the Mac stub's synthetic readings, and they are
identical to the PR #24 numbers row-for-row. The two columns this
re-run actually moves are `wall_time_s` and `tail_p99_ms`, which are
host-side timings derived from `time.perf_counter()` calls wrapping the
real Orin Python process — those are real Orin numbers.

The implication for any future W11+ work that needs **real** Orin
energy/memory telemetry: a separate `scripts/weekN_ablation_orin.py`
script would have to be written that constructs a `TegrastatsDaemon`
directly and feeds it into the framework, bypassing the runner's
`_make_telemetry()`. That is out of scope for closing the W8 PR #24
deviation.
